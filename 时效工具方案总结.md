# 销售订单时效工具方案总结

## 1. 目标
- 构建一个可配置化的销售订单时效统计工具，覆盖客户经理审批、财务审核、商标确认等线性节点。
- 避免全量扫表：通过“订单统计池 + 定时增量任务”处理时效计算。
- 支持 AI 查询：前端可按订单类型和字段进行时效分析与问答。

## 2. 核心设计
- **流程可配置**：流程模板、节点顺序、SLA、关键节点、中断/退出规则均可配置。
- **线性推进**：上一个审批未完成，后一个节点不计算。
- **订单池机制**：仅池内订单参与计算，订单完结/终止后退出池子。
- **定时增量计算**：任务按批次处理池内订单，产出节点时效、订单达标率和异常原因。

## 3. 架构流程
1. 业务事件进入系统（创建订单、节点开始/结束、驳回、取消、关闭）。
2. 池管理器根据事件维护订单池状态（ACTIVE/PAUSED/INTERRUPTED/EXITED）。
3. 定时任务只拉取需要处理的池内订单并执行计算。
4. 计算引擎按流程配置做线性节点时效计算，落地明细和汇总。
5. 前端看板 + AI 查询层读取结果，支持指标分析与异常追踪。

## 4. 数据模型（建议）
- `order_pool`：订单池状态、当前节点、下次执行时间、退出原因。
- `process_config`：流程配置（节点顺序、SLA、关键节点、规则版本）。
- `order_node_runtime`：节点级时效明细（开始/结束/时长/是否超时/异常原因）。
- `order_sla_summary`：订单汇总（总时效、达标状态、异常标签）。
- `event_log`：原始事件日志（幂等去重、重放校验）。

## 5. 入池、出池、中断机制
- **入池**：订单创建或有新审批事件时进入/重新激活。
- **出池**：
  - 正常出池：全部关键节点完成；
  - 业务出池：取消、关闭、终止；
  - 风险出池：超长挂起、超限中断。
- **中断**：
  - 触发条件：长时间无事件、关键事件缺失、回退次数超阈值；
  - 处理方式：标记 `INTERRUPTED`，进入人工或自动补偿队列；
  - 恢复方式：新事件到达或补齐数据后自动恢复并重算。

## 6. 时效与达标口径
- 节点时效：`node_end_time - node_start_time`
- 订单总时效：`final_node_end_time - order_create_time`
- 节点达标：`duration <= node_sla_threshold`
- 订单达标：
  - 严格口径：关键节点全部达标；
  - 比例口径：关键节点达标率达到阈值（如 >= 90%）。
- 支持自然时长与工作日历时长（节假日、班次）两种口径。

## 7. AI 查询方案
- **双通道路由**：
  - 简单指标问题走 NL2SQL（均值、分位、达标率、TopN）；
  - 复杂归因问题走 Agent 工具链（口径解释、跨维度对比）。
- **前端交互**：
  - 先选订单类型、字段、时间范围；
  - 再输入自然语言问题；
  - 返回图表、结论、口径说明和查询轨迹。
- **安全治理**：
  - SQL 模板白名单；
  - 字段级权限控制；
  - 敏感字段脱敏；
  - 查询审计留痕。

## 8. 分阶段落地
- **Phase 1（MVP）**：流程配置 + 订单池 + 定时增量 + 基础看板。
- **Phase 2**：中断恢复自动化 + 异常标签体系 + 工作日历口径。
- **Phase 3**：AI 双通道查询 + 权限审计 + 归因分析增强。

## 9. 验收标准
- 不全量扫表：仅处理池内待计算订单。
- 线性节点准确：前置未完成时后置节点不误算。
- 幂等与一致性：重复执行结果一致，失败可重试可补偿。
- 可运营：中断可追踪、可恢复，AI 查询可解释。
